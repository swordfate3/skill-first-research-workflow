# OCRA v4.7 架构思想分析

## 1. 这套系统本质上在做什么

OCRA v4.7 不是一个单纯的“论文总结器”，也不是一个只会调某个大模型 API 的壳。  
它的核心思想是：

**把密码学科研流程拆成可持续积累的几个阶段，并让多模型协同完成这些阶段。**

这几个阶段大致是：

1. 扫描新增论文 / 代码
2. 把原始材料转成结构化记忆
3. 基于记忆做碰撞创新
4. 基于碰撞生成研究方向
5. 把结果通过 Web Dashboard 和报告输出出来

也就是说，它真正想做的不是“一次性回答”，而是一个**长期运行的科研记忆-碰撞-方向生成系统**。

---

## 2. 系统的总设计思想

从代码和参考文档来看，这套系统有五个最核心的设计思想。

### 2.1 增量处理，而不是全量重跑

`scripts/scan_new.py` 通过 `registry.json` 记录：

- 文件路径
- 文件 hash
- 是否已经处理
- 对应 memory_id

这意味着它不是每次把所有论文重新喂给模型，而是只处理：

- 新文件
- 内容发生变化的文件

这个思路非常关键，因为科研资料会持续增加，如果每次全量分析，成本、耗时和重复工作都会失控。

一句话说，它把“论文处理”从一次性批处理，变成了**可持续增量同步**。

---

### 2.2 结构化记忆优先，而不是自然语言堆积

OCRA 的真正中枢不是 prompt，而是 `memory/` 里的结构化 JSON：

- `papers.json`
- `codes.json`
- `collisions.json`
- `directions.json`
- `registry.json`

对应设计在 `references/memory_schema.md` 里写得很清楚。

它要求模型输出的不是“自由文本摘要”，而是可程序消费的结构化条目，例如：

- 分类标签
- 核心贡献
- 方法学
- 实验结果
- 局限性
- `innovation_seeds`
- `connections`

其中最关键的是 `innovation_seeds`，因为这部分不是“总结过去”，而是在为下一步“碰撞创新”准备燃料。

也就是说，OCRA 的记忆系统不是知识库式归档，而是**面向创新生成的结构化中间层**。

---

### 2.3 多后端协同，但不是平均分工

这套系统虽然写着多 AI 协调，但它不是“每个模型都做同样的事”，而是有明显的职责划分。

从 `openclaw.py`、`orchestrator/router.py`、`ai_backends/*.py` 来看，职责大致是：

- **Claude API**
  - 适合处理 PDF 原文件
  - 支持 document 输入
  - 支持 Extended Thinking
  - 还承担架构图 / SVG / Mermaid 相关的生成

- **OpenAI API**
  - 适合处理提取后的长文本
  - 用 Responses API + background mode 做长时间推理
  - 负责提供第二视角或并行分析

- **Claude Code**
  - 更像本地执行和应急后备
  - 更适合代码分析、实现、调试
  - 当 API 不可用时才兜底某些流程

所以它的多后端协同，并不是“投票式 ensemble”，而是：

**按材料类型和任务类型，把最适合的模型放在最适合的位置。**

---

### 2.4 异步轮询优先，避免长连接任务不稳定

这是 v4.7 很明显的一个工程取向。

`ai_backends/openai_api.py` 里采用的是：

1. 提交 `Responses API` 请求
2. `background=true`
3. 保存 response id
4. 轮询直到 `completed`

`orchestrator/engine.py` 中的图表补充流程则采用：

1. 提交 Claude Messages Batch
2. 轮询 batch 状态
3. 结束后读取结果

这说明它非常在意一类现实问题：

- 推理耗时很长
- HTTP 长连接容易断
- 单次请求容易超时

所以它把“长推理”统一改造成了**提交任务 + 后台轮询**模式。  
这不是学术设计，而是非常实用的工程设计。

---

### 2.5 把“创新生成”拆成两级：碰撞 -> 方向

这是 OCRA 最像“科研助手”而不是“论文摘要器”的地方。

它不是直接从论文生成研究方向，而是拆成两步：

1. **碰撞（collision）**
   - 找两篇论文 / 论文与代码之间的交叉点
   - 形成创新假说

2. **方向（direction）**
   - 基于一批高价值碰撞
   - 再进一步抽象成更完整的研究问题、方法路径、评估方案

这个拆法很有价值，因为它把“创新”从一个模糊动作分成了两个更清晰的层次：

- 微观层：局部技术交叉
- 宏观层：可执行研究方向

这比直接问模型“给我几个创新点”要稳得多。

---

## 3. 系统数据流：它到底是怎么跑起来的

如果从运行路径看，OCRA 的完整数据流是下面这样。

### 阶段 1：初始化与扫描

入口在 `openclaw.py`：

- `ocra init`
- `ocra scan`
- `ocra analyze`

扫描逻辑在 `scripts/scan_new.py`：

- 建立 `workspace/papers`、`workspace/codes`、`workspace/memory`
- 通过 SHA-256 检测新增或修改
- 把结果写入 `registry.json`

这一阶段的核心不是分析，而是**确定哪些对象值得进入分析链路**。

---

### 阶段 2：论文 / 代码分析

论文分析有两条路线。

#### 论文分析

在 `orchestrator/engine.py` 中，核心方法是：

- `analyze_paper()`
- `analyze_paper_enhanced()`
- `analyze_paper_dual_api()`

其中真正体现 v4.7 思路的是 `analyze_paper_dual_api()`：

- Claude API 直接看 PDF 原文件
- OpenAI API 看提取后的文本
- 两边并行执行
- 取成功结果并记录双方参与

也就是说，它希望同时拿到：

- Claude 对图表、公式、版式的利用能力
- OpenAI 对长文本推理的稳定输出

#### 代码分析

代码分析通过：

- `scripts/parse_code.py`
- `engine.py -> analyze_code()`

其目标不是简单解释代码，而是提取：

- 架构
- 模块
- 优化技巧
- 安全措施
- 可复用模式

所以代码在 OCRA 里不是附件，而是和论文平行的一类知识源。

---

### 阶段 3：写入结构化记忆

分析结果会写入：

- `papers.json`
- `codes.json`

而不是只保存在日志里。

这一步意味着 OCRA 不是“调用模型 -> 打印答案”，而是：

**调用模型 -> 写入统一 memory schema -> 供后续阶段复用**

这就是它能做“持续科研工作流”的关键。

---

### 阶段 4：碰撞创新

这部分有两层。

#### 4.1 启发式定义层

`scripts/collision.py` 和 `references/collision_engine.md` 给出了碰撞哲学和启发式维度，例如：

- 跨领域迁移
- 差距填补
- 理论统一
- 组合创新

这些更像“碰撞规则库”。

#### 4.2 实际运行层

真正运行时，核心逻辑在 `orchestrator/engine.py -> run_collision()`。

实际过程是：

1. 从 `papers.json + codes.json` 合并成 `all_items`
2. 枚举候选 pair
3. 跳过已经记录在 `collisions.json` 里的 pair
4. 为每对构造 `build_collision_prompt()`
5. 并行发给 Claude API 和 OpenAI API
6. 若返回 `has_valid_collision=true`，则写入 `collisions.json`

这里有一个很重要的现实约束：

当前实现里只处理：

```text
new_pairs[:20]
```

也就是说，它已经意识到 pairwise collision 可能组合爆炸，所以在运行层直接做了一个**硬截断**。

这说明它目前对“碰撞规模问题”的处理方式还比较工程化，而不是算法化：

- 先限制数量
- 还没有做很强的候选召回 / Top-K 过滤

---

### 阶段 5：生成研究方向

方向生成在 `engine.py -> suggest_directions()`。

它的核心思路是：

1. 读取 `collisions.json`
2. 按评分排序
3. 每批 5 条碰撞
4. 双后端并行生成方向建议
5. 写入 `directions.json`

这里能看出它的另一个设计判断：

**方向生成不是逐条碰撞单独出结论，而是让一批碰撞共同作为上下文。**

这会让生成的方向更像“研究规划”，而不是单一点子。

同时它支持 `since_ts`，也就是：

- 只对新碰撞生成新方向

这又回到了它的总体风格：**增量化工作流**。

---

### 阶段 6：补图与展示

图表补充在：

- `engine.py -> supplement_diagrams()`
- `_diagram_claude_batch()`

这一步的设计很有意思：

- 主分析阶段不要求模型生成图
- 图作为独立后处理任务生成

这样做的好处是：

1. 不把“分析”与“展示”绑死在一个 prompt 里
2. 图失败不会阻塞分析结果落库
3. 可以后续单独重跑 `ocra diagram`

Dashboard 则是一个很轻量的读取层：

- `dashboard.py` 提供 JSON API
- `dashboard.html` 负责前端展示

它直接读 memory 文件，而不是再走数据库或服务端业务层。  
这体现出 OCRA 的另一个偏好：

**文件系统就是数据库，Dashboard 只是这个数据库的可视化外壳。**

## 4. 它最核心的“思想抽象”

如果把具体实现都拿掉，只留下思想，我认为 OCRA v4.7 的核心可以归纳成下面四句话。

### 4.1 科研对象先转成结构化记忆，再做创新

它不相信“直接在原始论文上做创新生成”是稳定的。  
所以先做 memory abstraction，再做 collision / direction。

这是它最重要的中间层思想。

### 4.2 创新不是凭空生成，而是从可碰撞的字段中生长出来

`innovation_seeds`、`connections`、`classification` 这些字段并不是装饰。  
它们是专门为了后续碰撞和方向生成设计的。

换句话说，OCRA 把“创新生成”前置到了数据结构设计里。

### 4.3 多模型不是为了热闹，而是为了视角互补和任务解耦

Claude 看 PDF 原件、OpenAI 跑后台长推理、Claude Code 做本地执行。  
这背后不是多模型崇拜，而是：

- 输入形态不同
- 推理时延不同
- 输出职责不同

它试图让不同模型各自干最擅长的那部分。

### 4.4 文件驱动比数据库驱动更轻量，更适合个人科研工作流

OCRA 没有引入复杂数据库、消息队列、任务中心。  
它把所有状态都存在 JSON 和目录结构里。

这让它：

- 易迁移
- 易备份
- 易人工检查
- 易直接给 Dashboard 消费

代价是：

- 一致性和并发能力有限
- 大规模场景扩展性一般

但对个人或小团队科研助手来说，这个权衡是合理的。

---

## 5. 这套系统的优点

### 5.1 很明确地把“分析”和“创新”拆开了

很多研究助手停留在摘要层，而 OCRA 走到了：

- 分析
- 记忆
- 碰撞
- 方向

这条链路更接近真实科研过程。

### 5.2 中间状态全可见

由于核心状态都在 JSON 文件里，所以你可以直接检查：

- 某篇论文到底被分析成什么样
- 某个碰撞为什么被保留
- 某个方向是基于哪些碰撞来的

这比“黑箱 agent 一次跑到底”更适合调试和迭代。

### 5.3 对长时任务的工程处理比较成熟

background polling、batch polling、supplement 模式、diagram 后处理，这些都说明它是按“真实使用中的不稳定性”来设计的，而不是只在短 demo 上成立。

### 5.4 有较强的增量思维

`--supplement` 几乎贯穿整个流程：

- 新论文补分析
- 新 pair 补碰撞
- 新碰撞补方向
- 缺图补图

这很适合持续往知识池里加材料。

---

## 6. 目前实现上的局限

### 6.1 碰撞组合控制还不够聪明

`run_collision()` 当前是枚举 pair 后只取前 20 对。  
这说明它已经意识到组合爆炸问题，但解决方式还是“截断”，不是“高质量召回”。

如果材料量变大，理想做法应该是：

- 先做候选筛选
- 再做 Top-K 碰撞
- 而不是简单截断

### 6.2 记忆 schema 很强，但也比较重

结构化字段很多，这提高了后续可用性，但也提高了模型输出负担。  
一旦模型输出 JSON 质量不稳定，整个链路都会受影响。

### 6.3 当前仍然偏文件型单机系统

适合个人科研工作流，但不太适合多用户并发、大规模任务队列或复杂权限管理。

### 6.4 某些“哲学层设计”与“运行层实现”还有缝隙

例如 reference 文档里谈了很多碰撞维度和哲学，但运行层的 pair 筛选与打分策略目前还比较直接。  
这说明它的设计愿景比当前算法实现更丰满。

---

## 7. 我对 OCRA v4.7 的整体判断

如果一句话总结：

**OCRA v4.7 的本质，是一个面向密码学科研的“结构化记忆驱动、多模型协同、增量式创新工作流”。**

它不是单点工具，而是一个小型研究操作系统：

- 输入层：论文 / 代码
- 中间层：memory schema
- 推理层：analysis / collision / direction
- 展示层：dashboard / report / diagram

它最有价值的地方，不在于“接了多少模型”，而在于它把科研流程中最难复用的部分做成了可积累、可追踪、可增量的中间层。

这也是它和普通“让 LLM 读论文”工具最大的区别。

---

## 8. 对当前 skill-first 项目的启发

如果是从这个项目中提炼可迁移思想，最值得继承的不是全部实现，而是下面几条：

1. **先做结构化 paper memory，再做 collision**
2. **把创新点生成拆成“碰撞”与“方向”两层**
3. **所有结果落盘成稳定中间状态，而不是只保留会话输出**
4. **支持 supplement / incremental 模式，而不是每次全量重跑**
5. **展示层和分析层解耦，图表作为后处理**

如果以后要做轻量版 skill-first 研究工作流，我会优先保留这些思想，而不是原样搬运它的整个 runtime。
